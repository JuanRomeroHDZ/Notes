Basándonos en el modelo OSI como guía para encontrar el fallo, existen distintas metodologías que podemos seguir para solucionar el problema:

**1. Bottom-Up**
Podemos iniciar abajo para arriba basandonos en el modelo OSI, verificar cosas fisicas, switches, routers y de ahí sigue para arriba.

**2. Top Down**
Se utiliza este método cuando el problema ya ha sido identificado. Generalmente, ocurre cuando existe algún inconveniente dentro del software, por lo que se recomienda aplicar este método para analizar la causa, documentar el problema y registrar la solución implementada.
![[Pasted image 20260616221754.png]]**3. Divide and Conquer**
Se suele utilizar este método cuando se desconoce la causa del problema. Consiste en analizar las diferentes capas del modelo OSI para identificar en qué nivel se encuentra la falla. Dependiendo de los resultados obtenidos, se puede continuar con otro método de solución de problemas o enfocarse directamente en la capa donde se detectó el inconveniente. Un ejemplo sería: si los usuarios no pueden acceder a una página web, pero sí pueden realizar un ping al servidor, el problema probablemente se encuentra en una capa superior a la capa 3, como la capa de transporte o aplicación. En cambio, si los usuarios no pueden realizar ping al servidor, la falla podría encontrarse en capas inferiores del modelo OSI, como la capa física, enlace de datos o red.

**4. Follow the Path**
Es uno de los métodos más sencillos, ya que consiste en seguir la ruta donde se presenta el problema desde su origen. Si el inconveniente comienza desde el usuario, se realiza un seguimiento del proceso hasta lograr recrear el problema y encontrar el punto donde ocurre la falla. El objetivo de este método es reducir la cantidad de enlaces y dispositivos involucrados en el diagnóstico, dejando únicamente aquellos elementos que podrían estar causando el conflicto y descartando los componentes que no tienen relación con el problema.

**5. Substitution**
Este método es conocido como el método de sustitución, ya que consiste en intercambiar un servicio, componente o dispositivo por otro que se encuentre en correcto funcionamiento. Si después del cambio el problema se soluciona, se determina que la causa estaba relacionada con el componente reemplazado; en caso contrario, se debe continuar investigando otras posibles causas. Este método funciona en casos específicos, como la sustitución de un router de borde o algún componente crítico de la infraestructura. Permite comprobar de manera rápida si el problema continúa o desaparece, aunque su aplicación puede ser limitada dependiendo de la disponibilidad de equipos o componentes de reemplazo.

**6. Comparison**
Este método, como su nombre lo indica, consiste en comparar un dispositivo o componente con otro que se encuentre funcionando correctamente para identificar posibles diferencias y determinar la causa del problema. Es útil cuando no se logra encontrar la causa raíz de la falla mediante otros métodos, ya que permite detectar configuraciones, comportamientos o elementos que puedan estar generando el inconveniente.

**7. Educated Guess**
Este método es menos estructurado, ya que depende principalmente de la experiencia y conocimiento de la persona encargada de realizar el diagnóstico. Se basa en la identificación de patrones, problemas conocidos y posibles soluciones a partir de situaciones similares que hayan ocurrido anteriormente. Es útil principalmente cuando el encargado cuenta con amplia experiencia, ya que permite encontrar soluciones de manera rápida; sin embargo, puede ser menos efectivo si no se tiene suficiente conocimiento del entorno o del problema.
